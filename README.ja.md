# cc-harness

[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)
[![Claude Code Plugin](https://img.shields.io/badge/Claude_Code-Plugin-orange)](https://github.com/Jinsong-Zhou/cc-harness)

**長時間アプリケーション開発のためのマルチエージェントハーネスフレームワーク（Claude Code プラグイン）**

Anthropic の研究に基づく：[長時間アプリケーション開発のためのハーネス設計](https://www.anthropic.com/engineering/harness-design-long-running-apps)

---

## なぜこのプラグインが必要か

単一エージェントによる複雑なコーディングタスクは、一見印象的でも実際には**壊れた**結果を生み出します。2つの根本的な問題：

1. **自己評価バイアス** — モデルは自分の平凡な作業を自信を持って称賛します。生成と評価を分離することが、品質向上の最も効果的な手段です。
2. **コンテキストの劣化** — 長時間のセッションでモデルは一貫性を失います。構造化されたハンドオフとコンテキスト管理により、数時間のビルドを軌道に乗せます。

本プラグインは、GAN に着想を得た**生成器-評価器アーキテクチャ**を実装し、反復的なフィードバックループによって真の品質を実現します。

## アーキテクチャ

```
ユーザープロンプト（1-4文）
        │
        ▼
┌──────────────┐
│   プランナー   │──→ SPEC.md（製品仕様書）
│  （読み取り専用）│
└──────────────┘
        │
        ▼
┌──────────────┐      ファイルベース通信       ┌──────────────┐
│  ジェネレーター │◄─────────────────────────►│  エバリュエーター│
│  （構築）       │                           │  （テスト）     │
└──────────────┘                             └──────────────┘
        │                                           │
   git commit                                Playwright/
        ▲                                  ブラウザでテスト
        └────────FAILなら反復──────────────
```

## クイックスタート

### インストール

```bash
# マーケットプレイスを追加
/plugin marketplace add Jinsong-Zhou/cc-harness

# プラグインをインストール
/plugin install cc-harness@cc-harness-marketplace
```

その後、Claude Code を再起動してください。

### 使い方

```bash
# ハーネス開発セッションを開始
/harness Web Audio API を使ったブラウザベースの DAW を構築

# いつでも現在の作業を評価
/evaluate
/evaluate ログインフロー
/evaluate フロントエンドデザイン

# セッションの進捗を確認
/harness-status
```

## コンポーネント

### エージェント

| エージェント | モデル | ツール | 目的 |
|------------|--------|--------|------|
| `harness-planner` | Opus | 読み取り専用 | 1-4文のプロンプトを製品仕様に変換 |
| `harness-generator` | Opus | 完全な書き込み | 機能を反復的に構築、git にコミット |
| `harness-evaluator` | Opus | 読み取り + Bash | 懐疑的な QA — Playwright/ブラウザでテスト |

### スキル

| スキル | トリガー | 目的 |
|--------|---------|------|
| `harness-loop` | 複雑なアプリの自律的構築 | プランナー → ジェネレーター → エバリュエーター ループの統合管理 |
| `context-management` | 長時間セッションの限界に接近 | 圧縮 vs リセット戦略、構造化ハンドオフ |
| `harness-tuning` | ハーネスパフォーマンスの最適化 | エバリュエーターの校正、コンポーネント削除、プロンプト工学 |

### コマンド

| コマンド | 目的 |
|---------|------|
| `/harness <プロンプト>` | ハーネス開発セッションを開始 |
| `/evaluate [範囲]` | スタンドアロン評価をトリガー |
| `/harness-status` | セッションの進捗とスコアを表示 |

### ルール

常時ロードされる規範的ガイドライン：
- **harness-workflow** — 複雑なビルドの必須パイプライン
- **evaluator-discipline** — QA における懐疑主義の優先
- **context-strategy** — コンテキスト管理のタイミングと方法
- **file-communication** — エージェント間のファイルベース通信プロトコル

### フック

| イベント | プロファイル | 目的 |
|---------|------------|------|
| `SessionStart` | minimal+ | ハーネス状態の初期化または復元 |
| `PreCompact` | minimal+ | コンテキスト圧縮前の状態保存 |
| `PostToolUse` | standard+ | 反復ログのためのファイル編集追跡 |
| `Stop` | minimal+ | メトリクス付きセッションサマリーの書き込み |

## 評価基準

### フルスタックアプリケーション

| 基準 | 重み | 評価内容 |
|------|------|---------|
| 製品の深さ | 高 | 実際の機能 vs ハリボテ — ユーザーはワークフローを完了できるか？ |
| 機能性 | 高 | 実際に動作するか？バグ、エッジケース、エラー状態？ |
| ビジュアルデザイン | 中 | 洗練された一貫性のあるフルビューポート UI |
| コード品質 | 低 | 基本能力チェック — 根本的な問題のみ |

### フロントエンドデザイン

| 基準 | 重み | 評価内容 |
|------|------|---------|
| デザイン品質 | 高 | 統一された全体 vs パーツの寄せ集め |
| 独創性 | 高 | 独自の設計判断 vs AI パターン / テンプレートデフォルト |
| 技巧 | 中 | タイポグラフィ階層、間隔、配色調和、コントラスト |
| 機能性 | 中 | 見た目に依存しないユーザビリティ |

## コスト参考

| アプローチ | 所要時間 | コスト | 品質 |
|-----------|---------|--------|------|
| 単一エージェント（ハーネスなし） | ~20分 | ~$9 | コア機能が壊れている |
| フルハーネス（スプリント付き） | ~6時間 | ~$200 | 動作する、洗練された |
| 簡略化ハーネス（スプリントなし） | ~4時間 | ~$125 | 動作する、同等品質 |

> フルハーネスは単一エージェントの ~20倍のコストですが、**実際に動作する**アプリケーションを生み出します。

## 新しいモデルへの適応

ハーネスの各コンポーネントは、モデルの限界に関する仮定を符号化しています。`harness-tuning` スキルを使用して、どのコンポーネントがまだ必要かを体系的に監査してください：

| コンポーネント | Opus 4.6 での状態 |
|--------------|-------------------|
| プランナー | **保持** — モデルは自然にスコープを野心的に拡張しない |
| エバリュエーター | **保持** — 自己評価バイアスは全バージョンで持続 |
| スプリント分解 | **削除** — モデルは 2 時間以上のビルドで一貫性を維持 |
| コンテキストリセット | **削除** — 最近のモデルにコンテキスト不安なし |
