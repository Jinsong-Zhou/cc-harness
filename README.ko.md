# cc-harness

[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)
[![Claude Code Plugin](https://img.shields.io/badge/Claude_Code-Plugin-orange)](https://github.com/Jinsong-Zhou/cc-harness)

**장시간 애플리케이션 개발을 위한 멀티 에이전트 하네스 프레임워크 (Claude Code 플러그인)**

Anthropic 연구 기반: [장시간 애플리케이션 개발을 위한 하네스 설계](https://www.anthropic.com/engineering/harness-design-long-running-apps)

## 왜 이 플러그인이 필요한가

단일 에이전트로 복잡한 코딩 작업을 수행하면 겉보기에는 인상적이지만 실제로는 **고장난** 결과를 만듭니다. 두 가지 근본적인 문제:

1. **자기 평가 편향** — 모델은 자신의 평범한 작업을 자신 있게 칭찬합니다. 생성과 평가를 분리하는 것이 품질 향상의 가장 강력한 수단입니다.
2. **컨텍스트 퇴화** — 장시간 세션에서 모델은 일관성을 잃습니다. 구조화된 핸드오프와 컨텍스트 관리가 수 시간의 빌드를 궤도에 유지합니다.

이 플러그인은 GAN에서 영감을 받은 **생성기-평가기 아키텍처**를 구현하여 반복적인 피드백 루프를 통해 진정한 품질을 달성합니다.

## 아키텍처

```
사용자 프롬프트 (1-4문장)
        │
        ▼
┌──────────────┐
│   플래너      │──→ SPEC.md (제품 사양서)
│  (읽기 전용)   │
└──────────────┘
        │
        ▼
┌──────────────┐       파일 기반 통신          ┌──────────────┐
│  제너레이터    │◄──────────────────────────►│   평가기       │
│  (빌드)       │                            │  (테스트)      │
└──────────────┘                             └──────────────┘
        │                                           │
   git commit                                Playwright/
        ▲                                  브라우저로 테스트
        └────────FAIL이면 반복──────────────
```

## 빠른 시작

### 설치

```bash
# 마켓플레이스 추가
/plugin marketplace add Jinsong-Zhou/cc-harness

# 플러그인 설치
/plugin install cc-harness@cc-harness-marketplace
```

그런 다음 Claude Code를 재시작하세요.

### 사용법

```bash
# 하네스 개발 세션 시작
/harness Web Audio API를 사용한 브라우저 기반 DAW 구축

# 언제든지 현재 작업 평가
/evaluate
/evaluate 로그인 흐름
/evaluate 프론트엔드 디자인

# 세션 진행 상황 확인
/harness-status
```

## 구성 요소

### 에이전트

| 에이전트 | 모델 | 도구 | 목적 |
|---------|------|------|------|
| `harness-planner` | Opus | 읽기 전용 | 1-4문장 프롬프트를 제품 사양으로 변환 |
| `harness-generator` | Opus | 전체 쓰기 | 기능을 반복적으로 구축, git에 커밋 |
| `harness-evaluator` | Opus | 읽기 + Bash | 회의적인 QA — Playwright/브라우저로 테스트 |

### 스킬

| 스킬 | 트리거 | 목적 |
|------|--------|------|
| `harness-loop` | 복잡한 앱 자율 구축 | 플래너 → 제너레이터 → 평가기 루프 통합 관리 |
| `context-management` | 긴 세션이 한계에 접근 | 압축 vs 리셋 전략, 구조화된 핸드오프 |
| `harness-tuning` | 하네스 성능 최적화 | 평가기 교정, 구성 요소 제거, 프롬프트 엔지니어링 |

### 명령어

| 명령어 | 목적 |
|--------|------|
| `/harness <프롬프트>` | 하네스 개발 세션 시작 |
| `/evaluate [범위]` | 독립 평가 트리거 |
| `/harness-status` | 세션 진행 상황과 점수 표시 |

### 규칙

항상 로드되는 규범적 가이드라인:
- **harness-workflow** — 복잡한 빌드의 필수 파이프라인
- **evaluator-discipline** — QA에서 회의주의 우선
- **context-strategy** — 컨텍스트 관리 시기와 방법
- **file-communication** — 에이전트 간 파일 기반 통신 프로토콜

### 훅

| 이벤트 | 프로필 | 목적 |
|--------|--------|------|
| `SessionStart` | minimal+ | 하네스 상태 초기화 또는 복원 (SPEC.md 또는 harness/ 존재 시에만) |
| `PreCompact` | minimal+ | 컨텍스트 압축 전 상태 저장 |
| `Stop` | minimal+ | 메트릭이 포함된 세션 요약 작성 |

## 평가 기준

### 풀스택 애플리케이션

| 기준 | 가중치 | 측정 내용 |
|------|--------|----------|
| 제품 깊이 | 높음 | 실제 기능 vs 외관 — 사용자가 워크플로우를 완료할 수 있는가? |
| 기능성 | 높음 | 실제로 작동하는가? 버그, 엣지 케이스, 에러 상태? |
| 비주얼 디자인 | 중간 | 세련되고 일관된 풀 뷰포트 UI |
| 코드 품질 | 낮음 | 기본 역량 확인 — 근본적 문제만 |

### 프론트엔드 디자인

| 기준 | 가중치 | 측정 내용 |
|------|--------|----------|
| 디자인 품질 | 높음 | 통일된 전체 vs 부품 모음 |
| 독창성 | 높음 | 독자적 디자인 결정 vs AI 패턴 / 템플릿 기본값 |
| 기교 | 중간 | 타이포그래피 계층, 간격, 색상 조화, 대비 |
| 기능성 | 중간 | 외관과 무관한 사용성 |

## 비용 참고

| 접근 방식 | 소요 시간 | 비용 | 품질 |
|-----------|---------|------|------|
| 단일 에이전트 (하네스 없음) | ~20분 | ~$9 | 핵심 기능 고장 |
| 전체 하네스 (스프린트 포함) | ~6시간 | ~$200 | 작동하며 세련됨 |
| 간소화 하네스 (스프린트 없음) | ~4시간 | ~$125 | 작동하며 동등 품질 |

> 전체 하네스는 단일 에이전트보다 ~20배 비용이 들지만 **실제로 작동하는** 애플리케이션을 만듭니다.

## 새 모델 적응

하네스의 각 구성 요소는 모델 한계에 대한 가정을 인코딩합니다. `harness-tuning` 스킬을 사용하여 어떤 구성 요소가 여전히 필요한지 체계적으로 감사하세요:

| 구성 요소 | Opus 4.6 상태 |
|-----------|--------------|
| 플래너 | **유지** — 모델이 자연스럽게 범위를 야심차게 확장하지 않음 |
| 평가기 | **유지** — 자기 평가 편향이 모든 버전에서 지속 |
| 스프린트 분해 | **제거** — 모델이 2시간 이상 빌드에서 일관성 유지 |
| 컨텍스트 리셋 | **제거** — 최근 모델에 컨텍스트 불안 없음 |
